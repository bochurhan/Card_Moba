## ✅ Phase 1 — 结算引擎（已完成）

> 目标：建立 C/S 共享的权威结算核心，确保确定性与可测试性。

| 模块 | 状态 | 说明 |
|------|------|------|
| `SettlementEngine` 四层结算 | ✅ | Layer 0/1/2/3 + 子阶段全部实现 |
| `BattleContext` 数据结构 | ✅ | `PlayerBattleState`、`LaneState`、`PlayedCard` 完整 |
| `SeededRandom` 确定性随机 | ✅ | Fisher-Yates 洗牌，C/S 结果一致 |
| `IEffectHandler` + 29 种效果 | ✅ | `HandlerRegistry` 全部注册 |
| `TargetResolver` 目标解析 | ✅ | 12 种 `EffectRange` 全覆盖 |
| `BuffManager` + `TriggerManager` | ✅ | Buff/Trigger 协作架构，孤儿触发器检测 |
| `DamageHelper` 统一伤害入口 | ✅ | 含无敌/护盾/减免/触发器全管道 |
| `RoundManager` 回合状态机 | ✅ | `InitBattle → PlayCard → EndRound` 主流程 |
| 架构风险清理 R-01~R-11 | ✅ | 高风险项全部修复，低风险记录在 TODO |

**里程碑**：`Shared/BattleCore` 可独立编译，无 `UnityEngine` 依赖，可接服务端。

## 📋 Phase 3 — 分路系统

> 目标：实现多路并行战斗，引入换路、支援、决战合并机制。

### 3.1 分路核心

| 任务 | 说明 |
|------|------|
| `LaneState` 完整实现 | 双方控制权、占领判定、分路得分 |
| 换路申请与确认流程 | 申请 → 等待对方确认 → 执行或拒绝 |
| 分路内 AI 策略扩展 | 按分路压力/血量自动决策换路 |
| `CrossLane` 效果目标解析 | `CardTag.CrossLane` 卡牌跨路选中对手 |

### 3.2 死亡与支援

| 任务 | 说明 |
|------|------|
| 死亡复活机制 | 死亡等待 N 回合，可被支援加速 |
| 死亡支援 | 友方可支援濒死路位，分担伤害 |
| 死亡惩罚（换路限制、下路输出限制） | 设计待定 |

### 3.3 决战阶段

| 任务 | 说明 |
|------|------|
| 分路得分转化决战优势 | 分路优势转化为决战 Buff/人数 |
| 决战分路合并逻辑 | 三路合并为一路，地图收缩 |
| 中枢塔BOSS前置战 | 触发决战前的最终分路对抗 |

**里程碑**：三路地图可正常运行，换路和决战流程可完整走通。

## 📋 Phase 4 — 网络对战

> 目标：支持真实玩家 1v1 实时联机对战，服务端权威结算。

### 4.1 网络协议层

| 任务 | 说明 |
|------|------|
| SignalR Hub 定义（`BattleHub`） | 连接、断开、重连处理 |
| 消息协议定义（`Shared/Protocol/`） | `CardPlayRequest`、`RoundStateUpdate` 等 |
| 心跳与超时机制 | 防止掉线导致对局卡死 |

### 4.2 服务端权威

| 任务 | 说明 |
|------|------|
| `Server/` 接入 `Shared/BattleCore` | 服务端作为唯一结算权威 |
| 回合操作锁定与超时自动提交 | 倒计时到 0 时服务端代为提交 |
| 战斗状态广播 | 每回合结算完成后广播 `BattleStateSnapshot` |

### 4.3 客户端同步

| 任务 | 说明 |
|------|------|
| 客户端接收状态快照 → 驱动 UI | 不在客户端跑结算，只做展示 |
| 动画序列器（按事件流播放动画） | `BattleEventRecorder` 事件流 → 动画队列 |
| 对战房间匹配（简单） | 创建/加入房间，2 人就绪开始 |

**里程碑**：两台设备可通过局域网/公网完成一局完整的 1v1 联机对战。

## 🔮 Phase 5 — 内容扩展

> 目标：丰富卡牌池、英雄系统、中枢塔，达到可测试游戏完整度。

### 5.1 卡牌内容

| 卡牌编辑器完善 | Unity 内 Inspector 直接配牌 |
| 卡牌插图接入 | Sprite 资源 + 卡面 UI |

### 5.2 英雄系统

| 任务 | 说明 |
|------|------|
| `HeroConfig` 完整定义 | 初始血量/护甲/能量/技能 |
| 英雄被动技能 | 通过 TriggerManager 注册被动触发器 |
| 英雄选择界面 | 开局前英雄选择/禁用流程 |
| 3 个初始英雄 | 基础测试用英雄（坦克/刺客/法师） |

### 5.3 中枢塔 (BOSS)

| 任务 | 说明 |
|------|------|
| 小怪配置与 AI | 决战前小怪波次，可通过卡牌击败 |
| BOSS 多阶段战斗 | 血量阶段触发不同技能模式 |
| BOSS 奖励系统 | 击杀 BOSS 获得稀有卡牌/Buff |
| 商店系统（可选） | 回合间隙购买卡牌 |

### 5.4 遗物系统

| 任务 | 说明 |
|------|------|
| `RelicConfig` 定义 | 遗物数据模型 |
| 遗物触发器注册 | 通过 TriggerManager 注册（Priority 200-499） |
| 初始遗物 10 件 | 覆盖主要触发时机 |

**里程碑**：可组织 5 人内部测试，收集完整游戏体验反馈。

## ⚠️ 关键路径与风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 卡牌数据设计滞后 | 无内容可测试，Phase 2 无法验证 | 优先设计 30 张基础牌，内容先行 |
| 分路 AI 难度 | Phase 3 AI 策略复杂，可能拖期 | 先用随机出牌 AI 走通流程，策略 AI 后补 |
| 网络同步复杂性 | Phase 4 重连、延迟补偿是已知难点 | 先局域网测试，公网优化后期处理 |
| 美术资源缺失 | UI 开发受阻 | 先用纯文字/占位图，美术资源并行跟进 |
