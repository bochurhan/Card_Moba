## 一、游戏开发的核心分层思想

### 1.1 为什么要分层？

想象一下建造一栋大楼：
- **地基** = 基础数据和协议（不依赖任何东西）
- **承重墙** = 核心业务逻辑（只依赖地基）
- **房间装修** = 用户界面（依赖承重墙，但承重墙不关心装修）

分层的好处：
1. **可维护性**：修改 UI 不会影响游戏规则
2. **可测试性**：可以单独测试核心逻辑，不需要启动 Unity
3. **前后端共用**：同一套规则代码可以同时运行在客户端和服务器
4. **团队协作**：不同人可以同时开发不同层

### 1.2 Card_Moba 的三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Presentation 层                          │
│         （UI显示、特效、音效、用户输入、场景管理）              │
│                 "用户看到和操作的一切"                         │
├─────────────────────────────────────────────────────────────┤
│                    GameLogic 层                              │
│         （业务流程控制、状态管理、事件分发）                    │
│              "连接 UI 和核心引擎的桥梁"                        │
├─────────────────────────────────────────────────────────────┤
│                    BattleCore 层                             │
│         （游戏规则、结算引擎、回合流程、数据模型）               │
│              "游戏的心脏，纯逻辑无渲染"                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                  Protocol & ConfigModels
                  （协议枚举、配置数据模型）
                    "所有层的公共语言"
```

## 三、核心模块详解

### 3.1 Shared/BattleCore — 战斗核心引擎

**职责**：实现所有游戏规则，是"游戏的心脏"。

**关键特性**：
- ✅ **纯 C# 代码**：不依赖 UnityEngine，可以在任何 .NET 环境运行
- ✅ **确定性**：相同输入必定产生相同输出（重要：用于前后端校验）
- ✅ **前后端共用**：同一份代码，客户端预测+服务器裁决

#### 核心文件解析

| 文件 | 职责 | 关键概念 |
|------|------|----------|
| `Context/BattleContext.cs` | 存储一整局对战的**所有可变状态** | 回合数、玩家列表、待结算操作、日志 |
| `Context/PlayerBattleState.cs` | 存储单个玩家的**所有运行时数据** | HP、能量、手牌、牌库、弃牌堆 |
| `Context/CardAction.cs` | 描述一次出牌操作 | 谁打、打谁、用什么牌 |
| `RoundStateMachine/RoundManager.cs` | **回合流程总控**，管理7阶段流转 | 初始化、出牌、结束回合、下一回合 |
| `Settlement/SettlementEngine.cs` | **效果结算器**，执行卡牌效果 | 瞬策即时结算、定策按优先级结算 |

#### 数据流示意图

### 3.2 Shared/Protocol — 通信协议

**职责**：定义客户端和服务器之间的"公共语言"。

**为什么需要单独一层？**
- 前后端需要用相同的枚举值、消息结构
- 修改协议时，两端可以同步更新
- 避免魔法数字（如用 `CardTrackType.瞬策牌` 代替数字 `1`）

#### 当前已定义的枚举

```csharp
// 卡牌双轨类型
public enum CardTrackType
{
    瞬策牌 = 1,  // 打出即结算
    定策牌 = 2,  // 回合末结算
}

// 卡牌标签（Flags 枚举，可组合多个标签）
// ⚠️ 注意：CardSubType 已被废弃并合并到 CardTag
[Flags]
public enum CardTag
{
    None = 0,

    // === 功能用途标签（用于筛选、搜索、UI 分类）===
    Damage    = 1 << 0,    // 伤害型
    Defense   = 1 << 1,    // 防御型
    Counter   = 1 << 2,    // 反制型
    Support   = 1 << 3,    // 辅助型
    Control   = 1 << 4,    // 控制型
    Draw      = 1 << 5,    // 抽牌型
    Legendary = 1 << 6,    // 传说卡

    // === 行为修饰标签（影响卡牌结算行为）===
    CrossLane = 1 << 10,   // 跨路生效
    Exhaust   = 1 << 11,   // 消耗（使用后移出对局）
    Cycle     = 1 << 12,   // 循环（弃牌堆回到牌库）
    Execute   = 1 << 13,   // 斩杀（对低血量目标额外效果）
}

// 卡牌目标类型
public enum CardTargetType
{
    Self = 1,           // 自己
    SingleEnemy = 2,    // 单个敌人
    SingleAlly = 3,     // 单个友方
    // ... 更多
}
```

> **架构说明**：`CardTag` 仅用于**元数据**（筛选、UI显示、条件检测），**不决定结算顺序**。  
> 结算优先级由 `CardEffect.EffectType` 数值范围决定（100=反制层，200=防御层，300=增减益层，400=伤害层）。

### 3.3 Shared/ConfigModels — 配置数据模型

**职责**：定义"配置表"的数据结构。

**关键区分**：
- **ConfigModels** = 静态配置，运行前就确定（如：火球术造成4点伤害）
- **Context** = 运行时状态，游戏中实时变化（如：玩家当前HP=15）

```csharp
// CardConfig.cs — 一张卡牌的所有配置信息
public class CardConfig
{
    public int CardId { get; set; }           // 唯一ID
    public string CardName { get; set; }      // 显示名称
    public string Description { get; set; }   // 描述文本
    
    public CardTrackType TrackType { get; set; }    // 瞬策/定策
    public CardTag Tags { get; set; }               // 功能标签（Flags，可组合）
    public CardTargetType TargetType { get; set; }  // 作用目标
    
    public int EnergyCost { get; set; }       // 能量消耗
    public int Rarity { get; set; }           // 稀有度
    
    public List<CardEffect> Effects { get; set; }   // 效果列表
    
    // ── 便捷属性 ──
    public bool IsLegendary => HasTag(CardTag.Legendary);
    public bool HasTag(CardTag tag) => (Tags & tag) != 0;
}
```

> **架构变更记录**（V1.1）：  
> - `SubType` 属性已被移除，功能合并到 `Tags` (Flags 枚举)  
> - 新增 `Effects` 列表，支持一张卡配置多个效果  
> - 效果的 `EffectType` 决定结算层，而非 `Tags`

### 3.4 Client/GameLogic — 客户端逻辑层

**职责**：作为 UI 层和 BattleCore 的"中间人"。

**为什么不直接让 UI 调用 BattleCore？**
1. BattleCore 是纯逻辑，不知道 Unity 的存在
2. 需要有人处理 AI、网络、事件通知
3. UI 不应该关心"怎么结算"，只关心"结果是什么"

### 3.5 Client/Presentation — 表现层

**职责**：处理所有"用户能看到和操作的东西"。

**关键原则**：
- ❌ **不做逻辑判断**：不决定"火球术该造成多少伤害"
- ✅ **只做展示和输入**：显示伤害数字、播放特效、响应点击

#### BattleUIManager 的职责

| 功能分类 | 具体内容 |
|----------|----------|
| **状态显示** | 刷新 HP、能量、护盾、手牌数量 |
| **手牌管理** | 生成卡牌 UI、绑定点击事件 |
| **用户输入** | 响应卡牌点击、结束回合按钮 |
| **日志面板** | 显示战斗日志、自动滚动 |
| **结束面板** | 胜利/失败/重新开始 |

### 4.2 依赖规则（铁律）

| 规则 | 说明 |
|------|------|
| ✅ 上层可以引用下层 | Presentation → GameLogic → BattleCore |
| ❌ 下层禁止引用上层 | BattleCore ❌→ GameLogic |
| ❌ 同层禁止互相引用 | Network ❌→ Data |
| ✅ 都可以引用 Protocol/ConfigModels | 这是公共语言 |
| ⚠️ BattleCore 禁止引用 UnityEngine | 保持纯 C#，前后端共用 |

## 七、常见问题 FAQ

### Q1: 为什么 BattleCore 不能用 UnityEngine？

**A**: 因为服务器端也需要运行同样的结算逻辑。服务器是用 .NET Core 跑的，没有 Unity 环境。如果 BattleCore 依赖 Unity，服务器就没法用了。

### Q2: 客户端和服务器都运行 BattleCore，那谁说了算？

**A**: 服务器说了算！
- 客户端：运行 BattleCore 做"预测"，让 UI 即时响应
- 服务器：运行 BattleCore 做"裁决"，是最终真相
- 如果预测错了：客户端收到服务器结果后，回滚到正确状态

### Q3: Presentation 层怎么知道状态变了？

**A**: 通过**事件（Event）**机制。
```csharp
// GameLogic 定义事件
public event Action OnStateChanged;

// Presentation 订阅
_gameManager.OnStateChanged += RefreshAllUI;

// GameLogic 触发
OnStateChanged?.Invoke();
```

### Q4: 为什么手牌每次都要销毁重建？

**A**: 这是最简单的实现方式（原型阶段优先）。未来优化时会改成**对象池**：
- 提前创建 N 个卡牌 UI
- 需要显示时从池中取
- 不需要时放回池中
- 避免频繁 Instantiate/Destroy

### Q5: 配置表在哪里？怎么读取？

**A**: 当前是硬编码在 `BattleGameManager.CreateTestDeck()` 里。后续计划：
1. 策划在 `Config/Excel/` 编写 Excel
2. 工具转换成 `Config/Json/` 的 JSON 文件
3. 客户端启动时从 `StreamingAssets/Config/` 加载
