# MatchFlow 说明

## 1. 分层目标

当前整局流程分为两层：

- `BattleCore`
  - 只负责单场战斗
- `MatchFlow`
  - 负责整局多场战斗、场间构筑、跨战斗继承

这样做的目的是避免把“整局逻辑”塞回单场战斗内核。

## 2. 当前整局模型

当前本地与服务端都围绕 `4+1` 模型运行：

- 前 4 场常规战斗
  - 单场 `1v1`
  - 每场 5 回合
- 第 5 场终局战斗
  - 当前仍按 `1v1` 测试
  - 回合上限 10

该模型是模板配置，不是写死在 `BattleCore` 中。

## 3. 关键运行时对象

### MatchContext

整局唯一上下文，持有：

- 当前阶段
- 当前战斗索引
- 玩家持久状态
- 队伍状态
- 当前活动战斗
- 当前构筑窗口

### MatchManager

整局状态机，负责：

- 开始对局
- 开始当前战斗
- 处理战斗结束
- 打开构筑窗口
- 推进下一场

### BuildWindowState

场间构筑窗口状态，负责：

- 当前机会序号
- 当前可选动作
- 升级候选
- 删牌候选
- 拿牌候选组
- 锁定状态

## 4. 当前构筑规则

支持动作：

- 回血
- 升级
- 删牌
- 拿牌

### 回血

- 普通情况恢复最大生命值的 30%
- 上一场战败则进入强制恢复
- 强制恢复时只能回血，回复 40%

### 拿牌

当前采用二段式：

1. 先承诺本次机会用于拿牌
2. 才揭示两组选牌
3. 每组可选 0 或 1 张
4. 跳过也会消耗本次机会

## 5. 装备触发

当前已接入最小装备被动模型。

已实现示例：

- `燃烧之血`
  - 每场战斗结束后回复 6 点生命

当前处理方式：

- 战斗开始时注册运行时
- 战斗结束后由 `MatchFlow` 回写持久状态

## 6. 与服务端的关系

当前服务端不会重写整局逻辑，而是直接托管：

- `Shared/MatchFlow`
- `Shared/BattleCore`

服务端职责是：

- 托管权威状态
- 接收命令
- 推进整局
- 生成快照
- 推送协议 DTO
